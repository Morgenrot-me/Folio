# 智能图像分类工具 — 数据模型设计

> 创建时间：2026-03-24
> 版本：v0.2（更新：确认混合文件夹方案）

---

## 总览

本地数据库采用 **SQLite**（通过 Flutter 的 `drift` 或 `sqflite` 访问），共五组核心数据表：

```
images              图片基础信息 + 特征数据
smart_folders       智能文件夹定义
folder_rules        智能文件夹规则树（AND/OR/NOT 逻辑）
clusters            相似图片聚类组
image_folder_map    图片与文件夹的归属关系（支持一图多夹）
```

---

## 一、images 表

> 每张图片对应一条记录，存储路径、元数据、全部提取特征。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | TEXT (PK) | 由文件路径生成的稳定 ID（SHA256 of path）|
| `file_path` | TEXT | 文件绝对路径 |
| `file_name` | TEXT | 文件名 |
| `width` | INTEGER | 图片宽度（px）|
| `height` | INTEGER | 图片高度（px）|
| `file_size` | INTEGER | 文件大小（bytes）|
| `taken_at` | INTEGER | 拍摄时间（EXIF，Unix 时间戳，可为 null）|
| `indexed_at` | INTEGER | 特征提取完成时间 |
| `phash` | INTEGER | 64 位感知哈希（用于相似图检测）|
| `semantic_vector` | BLOB | 512 维语义向量（float32 × 512，共 2048 bytes）|
| `is_screenshot` | INTEGER | 是否为截图（0/1）|
| `has_text` | INTEGER | 是否含文字（0/1）|
| `blur_score` | REAL | 模糊度评分（越高越清晰，拉普拉斯方差）|
| `dominant_hue` | REAL | 主色调（HSV 色相值 0~360）|
| `color_warmth` | REAL | 色温倾向（-1.0 冷 ~ +1.0 暖）|
| `cluster_id` | TEXT | 所属聚类 ID（FK → clusters.id，可为 null）|
| `gps_lat` | REAL | GPS 纬度（可为 null）|
| `gps_lon` | REAL | GPS 经度（可为 null）|

**设计要点：**
- `id` 用路径哈希而非自增整数，路径变动时可检测并更新，不产生重复记录
- `semantic_vector` 以 BLOB 存储原始 float32 字节，查询时在 Dart 层做余弦相似度计算
- 技术特征（截图/文字/模糊）与语义向量并列存储，便于规则引擎直接查询

## 二、smart_folders 表

> 用户创建的智能文件夹，每个文件夹对应一套规则树。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | TEXT (PK) | UUID |
| `name` | TEXT | 文件夹名称 |
| `icon` | TEXT | 图标标识符 |
| `color` | INTEGER | 文件夹颜色（ARGB）|
| `root_rule_id` | TEXT | 规则树根节点 ID（FK → folder_rules.id）|
| `sort_order` | INTEGER | 显示排序 |
| `created_at` | INTEGER | 创建时间 |
| `last_matched_at` | INTEGER | 上次规则匹配执行时间 |
| `export_path` | TEXT | 已导出到的系统文件夹路径（null = 未导出）|
| `export_mode` | TEXT | 导出模式：`COPY` / `MOVE`，null = 未导出 |
| `last_exported_at` | INTEGER | 上次导出时间（null = 未导出）|

## 三、folder_rules 表（核心）

> 使用**递归树结构**存储 AND/OR/NOT 逻辑组合，每条记录是树的一个节点。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | TEXT (PK) | UUID |
| `folder_id` | TEXT | 所属智能文件夹（FK）|
| `parent_id` | TEXT | 父节点 ID（null 表示根节点）|
| `node_type` | TEXT | 节点类型：`AND` / `OR` / `NOT` / `LEAF` |
| `feature_type` | TEXT | 仅 LEAF 节点：特征类型（见下表）|
| `comparator` | TEXT | 仅 LEAF 节点：比较符（见下表）|
| `value` | TEXT | 仅 LEAF 节点：比较值（JSON 序列化）|

### node_type 说明

| node_type | 含义 |
|-----------|------|
| `AND` | 所有子节点均为真 |
| `OR` | 任意子节点为真 |
| `NOT` | 子节点（只有一个）取反 |
| `LEAF` | 叶子节点，实际特征判断 |

### LEAF 节点的 feature_type

| feature_type | 说明 | 示例 value |
|--------------|------|------------|
| `IS_SCREENSHOT` | 是否截图 | `true` |
| `HAS_TEXT` | 是否含文字 | `true` |
| `BLUR_SCORE` | 模糊度 | `{"threshold": 50.0}` |
| `COLOR_WARMTH` | 色温 | `{"min": 0.3, "max": 1.0}` |
| `SEMANTIC_LABEL` | 语义标签 | `{"label": "outdoor", "threshold": 0.75}` |
| `CLUSTER` | 属于某聚类 | `{"cluster_id": "xxx"}` |
| `SIMILARITY` | 与某张图相似 | `{"image_id": "xxx", "threshold": 0.85}` |
| `GPS_REGION` | GPS 区域内 | `{"lat":39.9, "lon":116.4, "radius_km":5}` |

### 规则树示例

> 规则：`(截图 OR 含文字) AND NOT 模糊`

```
AND (根节点)
├─ OR
│   ├─ LEAF: IS_SCREENSHOT = true
│   └─ LEAF: HAS_TEXT = true
└─ NOT
    └─ LEAF: BLUR_SCORE < 30
```

对应数据库记录：

```
id=r1, parent=null,  type=AND
id=r2, parent=r1,    type=OR
id=r3, parent=r2,    type=LEAF, feature=IS_SCREENSHOT, comparator=EQUALS,     value="true"
id=r4, parent=r2,    type=LEAF, feature=HAS_TEXT,      comparator=EQUALS,     value="true"
id=r5, parent=r1,    type=NOT
id=r6, parent=r5,    type=LEAF, feature=BLUR_SCORE,    comparator=LESS_THAN,  value="30"
```

## 四、clusters 表

> 存储 HDBSCAN 聚类结果，每组相似图片形成一个聚类。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | TEXT (PK) | UUID |
| `name` | TEXT | 聚类名（自动生成或用户命名）|
| `centroid_vector` | BLOB | 聚类中心向量（512 维）|
| `image_count` | INTEGER | 成员图片数量 |
| `created_at` | INTEGER | 聚类生成时间 |
| `is_user_named` | INTEGER | 是否被用户手动命名（0/1）|

## 五、image_folder_map 表

> 记录图片与文件夹的归属关系，支持**一张图属于多个文件夹**。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | TEXT (PK) | UUID |
| `image_id` | TEXT | FK → images.id |
| `folder_id` | TEXT | FK → smart_folders.id |
| `source` | TEXT | 归属来源：`AUTO`（规则匹配）/ `MANUAL`（用户手动）|
| `assigned_at` | INTEGER | 归属时间 |
| `is_physical_primary` | INTEGER | 是否为物理主文件夹（0/1），每张图仅有一条记录可为 1 |

**设计要点：**
- `source` 区分自动与手动，允许未来对用户手动行为做学习反馈
- 用户手动移入/移出会覆盖或补充 AUTO 记录
- `is_physical_primary = 1` 表示文件已被物理移动到此文件夹的导出路径下；同一张图只能有一条主记录
- 其他文件夹的 `is_physical_primary = 0`，通过 `images.file_path`（已更新为新路径）继续访问文件

## 数据流总览

```
图片新增/变更
     │
     ▼
 特征提取（后台 WorkManager）
     │
     ├─ 写入 images 表（向量 + 技术特征）
     ├─ 更新 clusters 表（增量聚类）
     │
     ▼
 规则引擎扫描所有 smart_folders
     │
     ├─ 遍历 folder_rules 树 → 对 images 表执行特征匹配
     │
     └─ 写入 image_folder_map（source=AUTO）
```

## 混合文件夹方案说明（方案Y：主文件夹移动）

智能文件夹采用**虚拟多归属 + 单一物理主文件夹**的混合策略。

### 模式一：虚拟文件夹（默认，所有文件夹初始状态）

```
系统：DCIM/Camera/IMG_001.jpg     文件不移动
App：  [旅行] 和 [风景] 均显示此图   均为 App 内虚拟视图
```

- 零权限风险，只需 `READ_MEDIA_IMAGES`
- 一张图可同时属于任意多个文件夹
- 不破坏其他 App（微信、系统相册等）对文件的引用

### 模式二：设为物理主文件夹（用户主动触发移动）

```
操作：长按某个文件夹 → [设为物理主文件夹] → SAF 选择目标路径
结果：文件夹内所有图片 物理移动 到目标路径
效果：系统文件管理器可见；其余虚拟文件夹通过更新后的路径继续访问
```

**移动后状态：**

```
文件：Pictures/旅行/IMG_001.jpg   （已移动，原位不再存在）
images.file_path 更新为新路径
image_folder_map：
  [旅行].is_physical_primary = 1   ← 物理主文件夹
  [风景].is_physical_primary = 0   ← 仍可在 App 内看到此图（读新路径）
```

### 冲突规则

- 每张图只能有一个物理主文件夹（数据库唯一约束）
- 若图片已有物理主文件夹，再次触发其他文件夹的移动时：
  - 系统提示"该图已归属于 [旅行]"
  - 用户可选择：跳过 / 改变主文件夹（即将文件再次移动）

### Android 实现方案

```
1. 通过 SAF DocumentFile 获取目标文件夹写入权限
2. 对每张图片执行：
   a. DocumentFile.createFile() 在目标创建空文件
   b. OutputStream 写入内容（即 copy）
   c. 原文件删除（ContentResolver.delete）
   → 等效 MOVE，兼容所有 Android 版本
3. 更新 images.file_path = 新路径
4. 更新 image_folder_map.is_physical_primary = 1
5. 调用 MediaScannerConnection.scanFile(旧路径, 新路径)
   通知 MediaStore 刷新索引
```

> Android 10+ 可用 `ContentResolver.move()` 原子操作，但 SAF 写入方式兼容性更广。

### 权限需求

| 模式 | 所需权限 |
|------|----------|
| 虚拟文件夹 | `READ_MEDIA_IMAGES`（普通权限）|
| 物理主文件夹移动 | 只读 + SAF 目录选择器授权写入（用户主动授权）|
