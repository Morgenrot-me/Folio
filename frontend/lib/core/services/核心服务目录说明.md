# 核心服务层 (Services) 说明

## 1. 目录作用
`services/` 目录包含了整个 Flutter 前端应用的核心基础服务，负责协调数据库、AI 推理引擎、系统媒体扫描以及异步后台任务的调度。所有的业务逻辑和性能敏感操纵均下沉至本层统一处理。

## 2. 文件列表
- `app_settings_service.dart`: 负责应用层面的全局设置（偏好、开关等）存储与读取。
- `background_ai_worker.dart`: 借助 WorkManager 提供强韧的后台隔离执行环境 (Isolate)，在手机息屏状态下依然排队拉起特征提取任务。
- `cluster_service.dart`: AI 聚类服务，运行 DBSCAN 高级算法为万千图片做智能自动分类关联。
- `feature_extractor_service.dart`: **本项算力核心**。负责调用 C++ Skia 原生图层截取、Places365 / MobileCLIP 的 TFLite 模型并行处理计算、色调热量分析与 pHash 处理。已深度优化过单流管道并复用原始 256x256 收敛图，实现极致提速。
- `media_scanner_service.dart`: **媒体扫描主控**。两阶段解耦扫描架构（全量并行抓入照片数据库 + 异步挂载送交分析）。实现了底层事务 `database.batch` 的并发加速，大幅下降主线程物理 I/O 时间。
- `semantic_search_service.dart`: 提供语义脱机跨模态搜索功能。连接 SQLite R-Tree 执行高速浮点向量对比。
- `smart_folder_matcher_service.dart`: 当底层 AI 分析执行完毕时拦截并执行智能分组逻辑验证引擎。

## 3. 技术说明
在服务层我们实施了**大量高阶并发和多核并行压榨手段**（如在 `media_scanner_service.dart` 里的事务并发与预热查重）。所有模型流全由原生的 ByteBuffer 提供缓冲以求最大程度降低原生内存 (malloc) 分配过载的问题。

## 4. 使用示例
服务类实例均已被全局注册。
```dart
// 并发式扫描新图片的极其轻量的入口调用：
final count = await mediaScannerService.scanAndIndexMetadata((indexed, total) {
    print('当前扫描进度：$indexed / $total');
});
```

## 5. 扩展方式
如果将来要接入更多诸如 `Face Recognition`，请继续在该目录下新建专属 `Xxx_service.dart` 文件将业务脱离视图层逻辑抽取。

## 6. 注意事项
此处处理的 `photo_manager` 返回系统媒体内容属于严重涉及 I/O 与底层的敏感业务，**严禁在服务层增加任何同步 UI 的阻塞代码！**修改任何涉及到大批量遍历的代码前，务必谨慎考核对 Java Heap 与 Metaspace 可能引发的挤压。
